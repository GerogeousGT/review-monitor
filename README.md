# review-monitor

Мультиагентная система мониторинга отзывов для сервисного бизнеса. Изначально — домашнее задание курса Zerocoder (тема: автономные и многоагентные системы, CrewAI), собирается сразу как реальный пилотный кейс на примере фитнес-клуба.

**Пилот:** World Class, ул. Республики, 245, Тюмень — площадки 2ГИС, Яндекс Карты, Zoon.

## Что делает

- Собирает новые отзывы с нескольких площадок без повторного пересканирования всей истории (курсор по последнему увиденному отзыву)
- Определяет тональность **по темам внутри отзыва** (aspect-based), а не отзыва целиком — жалоба на бассейн внутри в целом позитивного отзыва не теряется
- Копит статистику в БД и поднимает персистентный жёлтый/красный алерт, если однотипных жалоб становится много за настраиваемое окно времени (не разовое уведомление — алерт живёт, пока не закрыт вручную)
- Следит за отзывами без ответа и напоминает, если SLA на ответ просрочен (сам определяет появление ответа — перечитывает страницу отзыва). Три эшелона по возрасту отзыва (`review_date`): до 90 дней — суточный watchdog, 90-180 дней — недельный обзор без срочного тона, старше 180 дней — нигде не всплывает, но остаётся `pending` в БД для статистики
- Пишет черновик ответа с учётом тона компании (документ или вытянутый из истории прошлых ответов) и присылает всё через Telegram — без автопостинга, решение всегда за человеком
- Раз в день шлёт сводку по всем открытым алертам (не только по изменившимся за сутки) — чтобы забытая проблема не потерялась
- Раз в неделю (понедельник утром) шлёт итоги для руководителя: объём отзывов за неделю и за месяц, топ хвалимого и топ ругаемого раздельно за месяц (за неделю обычно недостаточно данных — тема с 1 упоминанием не показывается как "топ"), список системного негатива (активные yellow/red алерты), просрочки по SLA за неделю
- Отдельно ловит "повторного негативщика" — один и тот же автор оставил несколько негативных отзывов НА ОДНОЙ площадке (author+platform, площадки не смешиваются — одинаковое имя на разных сайтах не гарантированно один человек). Порог кастомный под клиента (`alert_rules.repeat_offender` в конфиге). Напоминает раз в N дней, пока кто-то не отметит контакт с клиентом через `main_ack.py` — история (кто и когда связался) остаётся в БД для будущего дашборда

## Структура (multi-tenant с 2026-07-08)

Один код обслуживает несколько клиентов одновременно — общий движок, разные конфиги/БД/боты:

```
main_*.py          — точки входа (команды), запускаются напрямую или по cron/systemd
core/               — инфраструктура: db, config, env (выбор клиента), LLM-провайдер
agents/             — 4 из 5 компонентов архитектуры: Alert Engine, Sentiment
                       Analyst, Reply Strategist, Notifier
collectors/         — 5-й компонент (Collector), по файлу на площадку
db/schema.sql       — общая схема SQLite (не клиент-специфична)
clients/<slug>/     — ВСЁ клиент-специфичное: client_config.yaml, tone_of_voice.md,
                       db/reviews.db (своя SQLite), .env (свой Telegram-бот, свой
                       Apify-токен — разный биллинг). Ничего из этой папки не
                       переиспользуется между клиентами.
tests/              — pytest, ключевая логика (пороги алертов, дедуп, БД)
```

Выбор клиента — переменная окружения `CLIENT_SLUG` (обязательна, без неё любой
`main_*.py` падает с явной ошибкой, а не молча работает "не на том" клиенте):
```
CLIENT_SLUG=worldclass .venv/Scripts/python main_notify.py
```
или через обёртку: `./run_cycle.sh worldclass`.

- `clients/<slug>/client_config.yaml` — все настраиваемые под клиента параметры (площадки, теги, пороги алертов, SLA). Файл разбит на 3 блока: **КЛИЕНТ** (меняется под каждого нового заказчика), **ШАБЛОН ПОД НИШУ** (теги/категории — переиспользуется для той же отрасли), **ДВИЖОК** (технические дефолты, можно стартовать без изменений). Чек-лист разворачивания под нового клиента — в шапке самого файла.

## Новый клиент — быстрый разворот

Код общий, разворот — это просто новая папка `clients/<slug>/`, без копирования репозитория:
1. `mkdir clients/<новый_slug>` (+ `db/` создастся сама при первом запуске)
2. Скопировать `client_config.yaml` от похожего клиента, пройти чек-лист в его шапке
3. Написать/скопировать `tone_of_voice.md` под голос нового бренда
4. Завести `clients/<slug>/.env`: свой Telegram-бот (не переиспользовать чужого бота), свой `APIFY_API_TOKEN`
5. **Первый сбор истории — ОБЯЗАТЕЛЬНО с флагом:** `CLIENT_SLUG=<slug> python main_collect.py --backfill`. Без флага (или если позже понадобится доливать в БД старые отзывы вручную, в обход обычного цикла) следующий плановый `main_notify.py` разошлёт карточку по КАЖДОМУ вставленному отзыву одним пакетом — на практике это уже было 90 сообщений разом (см. CHANGELOG 2026-07-09)
6. Дальше — обычный цикл: `CLIENT_SLUG=<slug>` (или `./run_cycle.sh <slug>`) без флага

Уже развёрнуто: `clients/worldclass` (пилот, продакшн), `clients/daudelsport` (второй клиент, в процессе — см. PLAN.md).

Подробности — в [PLAN.md](PLAN.md).

## Установка

```
python -m venv .venv
.venv/Scripts/pip install -r requirements.txt
.venv/Scripts/python -m playwright install chromium
```

Корневой `.env` — fallback для ключей, реально общих на всех клиентов:
```
PROXYAPI_KEY=...   # активный провайдер (см. core/llm_provider.py), GROQ_API_KEY — запасной вариант
```

`clients/<slug>/.env` — per-client, обязателен перед запуском на этом клиенте:
```
TELEGRAM_BOT_TOKEN=...   # свой бот под клиента, НЕ переиспользовать бот другого клиента
TELEGRAM_CHAT_ID=...     # узнать через getUpdates после /start боту
APIFY_API_TOKEN=...      # console.apify.com → Settings → Integrations, нужен для сбора 2ГИС; свой на клиента (разный биллинг)
```

Все команды требуют `CLIENT_SLUG` в окружении. Дальше по порядку:
`main_collect.py` → `main_analyze.py` → `main_reply.py` → `main_alerts.py` → `main_notify.py`.
Раз в день (`digest_time` в конфиге, systemd timer) — `main_digest.py` и `main_watchdog.py`.
Раз в неделю — `main_weekly_stale.py` (просрочки 90-180 дней, см. `watchdog_recent_days`/`watchdog_stale_cutoff_days` в конфиге) и `main_weekly_summary.py` (итоги для руководителя: объём неделя/месяц, топ хвалимого/ругаемого за месяц, системный негатив = активные алерты, просрочки за неделю — понедельник утром).
Раз в `renotify_after_days` (конфиг, обычно раз в 2 дня) — `main_repeat_offender_notify.py` (напоминание по открытым repeat-offender алертам с кнопкой "Связался", молчит для acknowledged).
Каждые 5-15 минут — `main_repeat_offender_poll.py` (ловит нажатие кнопки через Telegram `getUpdates`; polling, не webhook — см. PLAN.md).
Пометить алерт "в работе" вручную (для тег-алертов, у repeat offender есть кнопка) — `main_ack.py`.
Обёртка для всего цикла сразу: `./run_cycle.sh <client_slug>`.

Тесты: `.venv/Scripts/python -m pytest tests/` (не трогают реальные клиентские БД — изолированная SQLite в памяти).

## Статус

- [x] **Collector** — Яндекс.Карты и Zoon через Playwright, 2ГИС через Apify (residential-прокси — собственный Playwright не пробивал антибот даже с жилого IP, см. PLAN.md), идемпотентная запись в БД, определение наличия ответа компании. Все три площадки проверены на реальных данных.
- [x] **Sentiment Analyst** — LLM, aspect-based тональность (тег ≠ тональность всего отзыва), управляемый словарь тегов с описаниями категорий (модель не плодит дубли), откалибровано вручную на 52 реальных отзывах.
- [x] **Alert Engine** — пороги из `client_config.yaml` (несколько окон на тег), персистентное состояние в БД (open/acknowledged/resolved, авто-закрытие при возврате к норме), Watchdog по просрочке SLA ответа. Ручное acknowledge — `main_ack.py` (CLI, без кнопок — см. PLAN.md). Проверено на синтетике (yellow/red/дедуп) и на реальном датасете (корректно — алертов нет, порогов никто не превысил).
- [x] **Notifier** — отправка в Telegram (свой бот @RewvieWClass_bot, не завязан на личные боты Жоржа): карточка нового отзыва (включая черновик ответа и заметку для менеджера), изменения алертов, дайджест просрочек по SLA. Backfill (первоначальная история) не спамит уведомлениями — уведомляются только по-настоящему новые отзывы.
- [x] **Reply Strategist** — черновик ответа + классификация (`positive`/`конструктив`/`эмоциональный`/`требует_проверки`) по гайду тона (`tone_of_voice.md`, собран на методичках OpenUp «Директор ФК»). Контекст активных алертов — только во внутреннюю заметку, не в публичный текст. Проверено на всех 51 отзыве, ожидающем ответа — 0 "требует_проверки" (в реальных данных нет троллинга/чёрного PR).

Все 5 компонентов архитектуры собраны и проверены на реальных данных.

- [x] **Деплой на VPS** — работает на боевом сервере, systemd-таймеры (пайплайн раз в 6 часов без watchdog,
  watchdog и дайджест ежедневно, дайджест в 18:00 МСК, недельный обзор давних просрочек — раз в неделю),
  без ручного запуска. Подробности — PLAN.md/CHANGELOG.md.
