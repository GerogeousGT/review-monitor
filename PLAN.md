# PLAN — review-monitor

Живой документ: что задумали, что сделали, что в очереди. Обновляется по ходу работы —
не только "после того как сделали", но и в момент, когда придумали что-то новое и решили
отложить. Цель — не терять идеи и не путать "мы это обсуждали" с "мы это построили".

## Что это

Мультиагентный мониторинг отзывов для сервисного бизнеса. Домашка курса Zerocoder
(автономные/многоагентные системы), собирается сразу как реальный демо-кейс на примере
фитнес-клуба (World Class, Тюмень).

**Стратегический разворот (2026-07-06):** не только пилот для одного клиента — план
докрутить до аналитики и быстро разворачивать на несколько компаний (прислать им готовый
анализ отзывов как демо/лид-магнит). Отсюда — приоритет на быстрое пересобирание конфига
под нового клиента (см. ниже) раньше, чем на Docker (Docker — когда клиентов станет 2-3+,
см. "Идеи").

## Архитектура (5 компонентов)

```
[Scheduler] → [Collector] → [Sentiment Analyst] → [Reply Strategist]
                                    ↓
                            [Alert Engine] → [Notifier → Telegram]
```

| Компонент | Файл | Что делает |
|---|---|---|
| Collector | `collectors/*.py`, `main_collect.py` | Playwright, курсор по последнему отзыву, не пересканирует историю |
| Sentiment Analyst | `agents/sentiment_analyst.py`, `main_analyze.py` | LLM, aspect-based тональность, управляемый словарь тегов+категорий |
| Alert Engine | `agents/alert_engine.py`, `main_alerts.py` | Пороги по окнам ("вспышка") + ступенчатая эскалация ("тлеющая проблема", 180→360→720 дней), персистентное состояние, Watchdog по SLA |
| Reply Strategist | `agents/reply_strategist.py`, `main_reply.py` | Черновик ответа + классификация, гайд тона (`tone_of_voice.md`) |
| Notifier | `agents/notifier.py`, `main_notify.py` | Telegram, свой бот @RewvieWClass_bot |
| Дайджест | `main_digest.py` | Ежедневная сводка: все открытые алерты + отзывы за сутки + просрочки |
| Acknowledge | `main_ack.py` | CLI: список алертов / пометить как "в работе" |

Инфраструктура (`core/`: db, config, env, llm_provider) и точки входа (`main_*.py`,
в корне) — см. "Структура кода" в README.md.

LLM-провайдер — `core/llm_provider.py`, единая точка переключения (сейчас ProxyAPI/gpt-4o-mini).

### Дашборд клиента — v1 (в разработке) и v2 (отложено)

Спроектирован на основе двух référence-документов от Жоржа (2026-07-14, сгенерированы
Perplexity — управленческая архитектура review-дашборда для руководителя сервиса).
Решение: v1 — только на данных, которые уже реально считаются в системе, никаких новых
аналитических функций/LLM-агентов. Всё, что требует новой логики — в v2, промониторить
использование v1 и решить, что реально нужно клиентам, а не проектировать вслепую.

**v1 (эта итерация), вкладка "Обзор":**
- Сводка сверху: открытые тег-алерты + repeat-offender алерты + просрочки по SLA
- Тренд отзывов за месяц по тональности (🟢🟡🔴)
- Открытые тег-алерты списком (тема, severity, счётчик, окно)
- Открытые repeat-offender алерты списком (автор, площадка, счётчик)
- Топ-3 хвалимых / топ-3 ругаемых темы за месяц (`get_top_tags_by_sentiment_since`)
- Просрочки: недавние (`get_overdue_reviews`) отдельно от "забытых" 90-180 дней
  (`get_stale_overdue_reviews`) — разный тон, не мешать в одну кучу
- Сравнение площадок (2ГИС/Яндекс/Zoon) — объём и доля негатива, новый лёгкий SQL-запрос
- **Скрытые проблемы** — отзывы с высокой оценкой (4-5★), но с негативным тегом внутри
  (aspect-based теги это уже хранят, только нужен новый запрос) — из документа Perplexity
  ("Sentiment vs Rating mismatch"), сильный аргумент именно ПРО aspect-based подход проекта

**v1, вкладка "Лента отзывов":** список отзывов с фильтрами (площадка/тональность/период),
пагинация, текст+теги+черновик ответа+статус.

**Правки после первого визуального ревью (2026-07-14):**
- Карточки сводки сверху не кликабельны (клик просто скроллил к тому же контенту ниже —
  бесполезная псевдо-интерактивность), сами карточки алертов/офендеров кликабельны и
  открывают модалку с drill-down до сырых отзывов (см. следующий пункт)
- Порядок секций: "Просрочки по SLA" сразу после "Открытых алертов" (было в самом низу)
- Подписи карточек — "Критично"/"Внимание" вместо "Красных алертов"/"Жёлтых алертов":
  цвет — деталь реализации, не смысл для владельца бизнеса

**v2 — отложено, требует новой разработки (не просто SQL-запрос к тому, что уже считается):**
- **Health Score (0-100)** — единая формула-агрегатор многих сигналов в одно число.
  Решить формулу отдельно, не с ходу — задача дизайна метрики, не кода
- **Auto-кластеризация жалоб без ручных тегов** (embeddings+clustering) — у нас уже есть
  управляемый словарь тегов через LLM, это другой подход поверх работающего — не очевидно
  нужен, конкурирует сам с собой
- **Root cause hypotheses / Executive Summary текстом** — новый LLM-вызов поверх кластера
  жалоб, отдельный агент, стоит денег на каждый запуск — не бесплатно, взвесить ценность
- **Anomaly detection** (всплеск за 24-72 часа, не 30/90 дней как burst сейчас) — новая
  метрика, у нас есть похожее (burst/chronic), но не такое короткое окно
- **Before/After sentiment recovery** — сопоставить дату ответа компании с тональностью
  последующих отзывов той же категории — отдельная аналитическая функция, которой нет
- **Emerging topics** (тема, которой не было месяц назад) — diff тегов между двумя
  периодами, несложно, но не реализовано
- **Детекция накрутки/ботов/дублей** — другая эвристика, чем repeat offender (паттерны
  текста, скорость публикации, отсутствие истории у автора) — не путать с уже готовым
  repeat offender, это разные задачи
- **VIP/LTV клиентов** — нет данных о ценности клиента вообще, потребовалась бы интеграция
  с CRM клиента — вне текущего скоупа продукта
- **Полная ролевая модель** (owner/service head/operational manager/branch manager/analyst
  из документов) — сейчас 2 роли (admin/client), под 5 ролей с разными срезами нужна
  куда более сложная модель прав. Делать, когда появится клиент с реальной оргструктурой,
  которой это нужно — оба текущих клиента (World Class, Даудель Спорт) однолокационные
- **Heatmap филиалы×темы** — оба текущих клиента одноточечные, `location_id` в схеме уже
  есть на будущее, но фича непроверяема на реальных данных, пока нет мультиточечного клиента
- **Sentiment vs Rating mismatch как отдельный AI-детектор** — v1 уже покрывает базовый
  случай через блок "Скрытые проблемы" (см. выше), развитая версия с AI-объяснением почему
  — v2
- **Иерархический дашборд по категориям тегов** (обсуждение 2026-07-14, после разбора
  ложного red-алерта "услуги" — см. CHANGELOG) — сейчас теги показываются плоским
  списком, категория (из `category_dictionary`) нигде не отображается, хотя в БД уже
  есть. Идея Жоржа: категория ("тренажёрный зал", "коммерция" и т.п.) — сворачиваемый
  верхний уровень с агрегированным счётчиком, разворачивается в конкретные теги внутри
  (частая проблема с широкими тегами — см. "услуги" — в том, что нельзя понять суть,
  не читая все отзывы; иерархия category→tag частично решает без AI-кластеризации,
  раз структура уже задана вручную в конфиге). Плюс график динамики жалоб по
  категории/тегу за месяц/год — сейчас нет вообще никакой временной визуализации,
  только текущий срез. Требует: новый агрегирующий SQL-запрос (группировка по
  category+tag с count), новый UI-компонент (collapsible tree или похожее), выбор
  библиотеки для графика (или простой SVG/canvas без зависимостей — не решено)

## Известные ограничения

- ~~2ГИС не собирается напрямую~~ — **решено.** Apify (`collectors/twogis.py`, actor `zen-studio/2gis-reviews-scraper`, residential-прокси) — проверено на реальных данных, 9 из 10 отзывов на 2ГИС уже с ответом клуба (в отличие от Яндекс.Карт и Zoon — полезный факт для питча). $1/1000 отзывов в выдаче, наш прогон на 10 отзывов стоил $0.017
- Официальный API 2ГИС не отдаёт текст отзывов вообще (только через отдел продаж, неясные условия) — не вариант
- 2ГИС в пользовательском соглашении (п. 2.13) прямо запрещает автоматический сбор данных — не проблема для внутреннего мониторинга своего клиента, но не афишировать механику, если это станет публичным продуктом
- На Яндекс.Картах и Zoon клуб почти не отвечает на отзывы (1 из 52) — но на 2ГИС всё наоборот, 9 из 10 с ответом. tone_of_voice по-прежнему гипотеза (мало реальных образцов даже с учётом 2ГИС), но неравномерность активности по площадкам — сама по себе полезный факт для питча
- "Чёрная реклама"/"троллинг" сознательно не классифицируются автоматически — риск ошибочно обидеть реального клиента выше пользы от автоматизации (см. `tone_of_voice.md`)
