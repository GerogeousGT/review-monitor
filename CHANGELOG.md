## [2026-07-20] — Фикс: "отмена" при правке тега закрывала всю модалку

Обнаружено сразу после предыдущей записи — кнопка "отмена" на форме правки
СУЩЕСТВУЮЩЕГО тега (клик ✎) делала `location.reload()`, скопировано как есть
с "Ленты отзывов", где это безобидно (та же страница). Внутри модалки
(drill-down по алерту/тегу/просрочке) `location.reload()` перезагружает весь
дашборд и закрывает модалку целиком — пользователь просто хотел отменить
правку, а не выйти из модалки. Кнопка "+ тег" такой проблемы не имела (уже
делала лёгкий локальный откат без reload) — баг был только в "отмене" правки
существующего чипа.

### Исправлено
- **`webapp/templates/dashboard.html: wireTagEditors`** — обработчик клика по
  ✎ вынесен в именованную функцию `wireEditButton(chip)` (была анонимная в
  `forEach`). Перед превращением чипа в форму сохраняется его исходный
  `innerHTML`; "отмена" теперь восстанавливает этот HTML на месте и
  перевешивает обработчик на новую (восстановленную) кнопку ✎ — без запроса
  на сервер, без перезагрузки страницы, модалка остаётся открытой. Save/delete
  по-прежнему делают `location.reload()` — там данные реально меняются,
  странице нужно обновиться.

### Проверено
- `pytest` — 86/86 (изменение чисто фронтенд). JS-синтаксис — `node --check`.
  Логика прослежена вручную: `wireEditButton` при восстановлении вешает
  обработчик на СВЕЖИЙ узел кнопки (старый уничтожен вместе со своим
  листенером при `chip.innerHTML = originalHTML`) — дублирования обработчиков
  нет.

## [2026-07-20] — Фикс: якорь не должен анимированно "съезжать" от верха

Продолжение предыдущей записи — Жорж проверил на проде: страница больше не
прыгает в начало, но теперь видно, как она анимированно "съезжает" сверху
вниз до якоря вместо мгновенного открытия сразу в нужном месте. Причина —
`scroll-behavior: smooth` на `<html>` (`webapp/templates/base.html`): при
переходе по URL с фрагментом (`#tag-categories`) после ПОЛНОЙ перезагрузки
браузер сначала рендерит страницу в позиции 0, затем анимированно скроллит
к фрагменту — это применяется и к самому первому скролле на загрузке, не
только к same-page переходам. В проекте нигде не было намеренной same-page
smooth-навигации, которая бы от этого зависела (проверено — единственное
использование правила было это самое, ничего не потеряно).

### Исправлено
- **`webapp/templates/base.html`** — `scroll-behavior: smooth` убран из
  правила `html`. Переход по якорю после перезагрузки теперь мгновенный —
  страница открывается сразу в нужном месте, без видимой анимации.

### Проверено
- Локально: рендер дашборда без ошибок, `scroll-behavior` в CSS отсутствует
  (только пояснительный комментарий на его месте). `pytest` — 86/86.

## [2026-07-17] — Дашборд: график динамики тональности (PLAN.md v2)

Первая фича из V2-бэклога дашборда после закрытия аудита (запрос Жоржа 2026-07-14:
статус-бар за фикс. 30 дней без истории → график с выбором периода).

### Добавлено
- **`webapp/charts.py`** — `render_diverging_bar_chart()`, inline SVG без внешних
  JS-библиотек (весь webapp server-rendered, ни одной CDN-зависимости в проекте).
  Форма выбрана по методичке dataviz-skill (`choosing-a-form.md`): тональность —
  "ordered-scale share" → **diverging stacked bar, centered on neutral**, не line
  chart. Колонки расходятся от нулевой линии: позитив вверх, негатив вниз,
  нейтраль — полоска поперёк нуля. Цвета — те же green/yellow/red, что уже
  используются в статус-баре дашборда и иконках Telegram-карточек (сознательно
  НЕ дефолтная палитра skill'а — не заводить второй визуальный язык тональности).
  Mark spec соблюдён: 4px закруглённый внешний край сегмента / острый у базовой
  линии, 2px surface-gap между сегментами стека, max_labels прореживает подписи
  при большом числе периодов.
- **Ось Y** (по ходу работы, запрос Жоржа) — "0" на базовой линии, круглое число
  (`_nice_ceiling`, лестница 1/2/5×10ⁿ) сверху и снизу. Единый масштаб на обе
  стороны — разные шкалы для позитива/негатива были бы визуальной ложью (бар "5"
  сверху и бар "5" снизу должны быть одной высоты).
- **HTML-тултип при наведении** (по ходу работы, запрос Жоржа: "на этой колонке
  4 отзыва") — заменил нативный SVG `<title>` (медленный, нестилизованный
  системный тултип) на позиционированный HTML-тултип, тот же паттерн vanilla JS,
  что уже был в файле для модалки drill-down по алерту. Текст: "15 июл — 4
  отзыва: 🟢 2 · 🟡 1 · 🔴 1" — с корректным русским склонением
  (`_plural_reviews`: отзыв/отзыва/отзывов). `data-tooltip`/`aria-label`
  экранированы через `_esc()`, JS читает через `textContent`, не `innerHTML`.
- **`core.db.get_review_sentiment_counts_by_period`** — агрегация тональности по
  периодам (день/неделя/месяц), Python-side группировка через `parse_review_date`
  (та же причина, что у остальных агрегатов в этом разделе — площадки отдают
  разные форматы дат). Пустые периоды заполняются нулями, не пропускаются —
  иначе диапазон без отзывов рвал бы ось X визуально. Шаг заполнения — всегда 1
  день (даже для week/month), чтобы не перескочить короткий месяц вроде февраля
  при фиксированном шаге.
- **Пресеты периода** (Неделя/Месяц/Квартал) — вкладки над графиком,
  `?period=` в URL. Гранулярность зависит от пресета: неделя/месяц — по дням,
  квартал — по неделям (иначе 90 дневных полосок на графике шириной с
  контентную колонку).
- **Свой диапазон дат** (по ходу работы, запрос Жоржа) — четвёртая вкладка,
  раскрывает форму с `<input type="date">` (обычный GET, без JS-фреймворка).
  Гранулярность теперь **автоматическая** для произвольного диапазона
  (`period.auto_granularity`: ≤35д → день, ≤180д → неделя, иначе месяц) — 3-летний
  custom-диапазон не пытается нарисовать 1000 дневных баров, сам уходит на
  месячную группировку. Защита от абсурдных значений: `until` в будущем
  обрезается до "сейчас", `since > until` — переставляются местами (не ошибка

## [2026-07-17] — Баг #1: алерт-пинги не доходили в Telegram (double-recompute)

### Найдено (аудит контекста + верификация на VPS/локально)
`run_cycle.sh` (=`ExecStart` основного systemd-сервиса на обоих клиентах) гоняет
`… → main_alerts.py → main_notify.py`. Оба вызывали `recompute_all`, а эта функция
и СЧИТАЕТ, и ЗАПИСЫВАЕТ состояние алертов. `main_alerts.py` пересчитывал первым
(применял diff, только печатал в journal), `main_notify.py` секундой позже
пересчитывал ПОВТОРНО — состояние уже актуально, второй diff пустой → **ни одно
сообщение об открытии/смене/закрытии алерта не уходило в Telegram**. Мониторинг
был не полностью слепой (дневной дайджест 18:00 всё равно перечисляет все открытые
алерты), но real-time пинг «пересечён порог по теме X» был мёртв.

Верификация до правки:
- VPS: `systemctl cat review-monitor-worldclass.service` → `ExecStart=run_cycle.sh`.
- Локально (синтетика): 1-й `recompute_all` → `opened`, 2-й подряд → `0 изменений`.
- Прод-БД (read-only): 4 открытых тег-алерта на worldclass (возврат средств/услуги/
  персонал/организация), которые пинга в Telegram в момент открытия не получали.

### Исправлено
- **Отправка алертов колокирована с единственным пересчётом** — в `main_alerts.py`
  (новая функция `send_alert_changes`, шлёт тонкий пинг на каждое изменение
  тег/зона-алерта; `notify` инъектируется для теста; каждая отправка в try/except,
  чтобы сбой доставки одного не ронял весь `run_cycle.sh` с `set -e`).
- `main_notify.py` больше НЕ пересчитывает алерты — только карточки новых отзывов
  (убраны `recompute_all` и импорт форматтеров алертов, `load_config` больше не нужен).
- Зональные алерты теперь тоже шлют тонкий пинг (раньше жили только в дашборде) —
  `format_zone_alert_message`/`format_zone_resolved_message` в `agents/notifier.py`
  (решение Жоржа 2026-07-17: «есть проблема с зоной такой-то → иди в дашборд»).
- Тег-алерт-пинги дополнены строкой «Разбор — в дашборде» — закрепляем разделение
  ответственности: **Telegram = сигнал + заголовок, дашборд = drill-down/аналитика**.
- Repeat offender НЕ входит в этот пинг — по-прежнему уведомляется отдельным таймером
  `main_repeat_offender_notify.py`.

### Проверено

  Яндекс.Картах) из подсчёта — иначе разные анонимные авторы ложно считались бы одним
  и тем же "повторным негативщиком"

### Решения
- **Кнопка "Связался с клиентом" в Telegram НЕ реализована, хотя код для неё написан
  и оставлен в проекте** (`agents/notifier.py`: `repeat_offender_ack_keyboard`,
  `get_updates`, `answer_callback_query`, `remove_message_keyboard`;
  `main_repeat_offender_poll.py` — не вызывается из systemd). Причина: получатель
  этого уведомления — сервисная служба клиента, не технарь, `main_ack.py` (CLI) для
  неё недоступен физически, нужна была кнопка. Реализовали через **polling**
  (периодический `getUpdates`, без нового долгоживущего процесса) — живой тест
  показал плохой UX: между нажатием кнопки и её исчезновением задержка до следующего
  опроса, пользователь не видит мгновенной реакции и не понимает, сработало ли
  ("нажал, ничего не произошло"). Правильное решение — **webhook** (Telegram сам
  дёргает HTTP-эндпоинт при нажатии, мгновенно), но это новый постоянно работающий
  процесс на VPS (Flask/gunicorn за nginx, как `landing-forms`) — сознательно
  отложено, делать вместе с доменом/лендингом/дашбордом (см. PLAN.md), не отдельным
  патчем ради одной кнопки
- **Временная мера вместо кнопки — жёсткий потолок `max_notifications`**: напоминание
  уходит `renotify_after_days` раз в `max_notifications` раз, потом молчит само
  (алерт остаётся `status='open'`, просто без дальнейших Telegram-уведомлений — виден
  в `main_ack.py`/дайджесте для ручного отслеживания). Не бесконечный спам без
  возможности остановить со стороны получателя
- При живом тестировании кнопки нашёлся и исправлен реальный баг: `answerCallbackQuery`/
  `editMessageReplyMarkup` падают с HTTP 400, если `callback_query_id` устарел (Telegram
  даёт ограниченное окно на ответ) или апдейт уже обработан повторно (Telegram иногда
  доставляет один callback дважды). Оба вызова обёрнуты в try/except в
  `main_repeat_offender_poll.py` — это только обратная связь в интерфейсе Telegram
  (всплывающая подсказка, скрытие кнопки), не бизнес-логика; `db.acknowledge_alert`
  к этому моменту уже сохранён, падать из-за косметики нельзя — иначе offset-курсор
  апдейтов не сохранится и тот же апдейт будет пытаться обработаться заново каждый прогон

## [2026-07-06] — Alert Engine + Watchdog

### Добавлено
- `alert_engine.py` — чистый код (без LLM): считает `COUNT(DISTINCT review_id)` с негативной тональностью по каждому тегу в каждом настроенном окне (`client_config.yaml → alert_rules`), берёт худшую сработавшую severity
- `main_alerts.py` — точка входа: пересчитывает алерты + Watchdog по просроченным SLA ответа
- В `db.py`: `alerts`-CRUD (create/update/resolve/get_active), `get_negative_tag_events`, `get_overdue_reviews`

### Решения
- Персистентность: один активный алерт на (тег, точка) — при повторном пересчёте либо обновляется severity на месте (не сбрасывая `status`, если уже `acknowledged`), либо создаётся новый, либо авто-закрывается (`resolved`), если счётчик в окне естественно опустился ниже жёлтого порога
- Дедупликация внутри отзыва (несколько упоминаний одного тега в одном тексте) не даёт задвоения счёта — проверено синтетическим тестом
- Watchdog отдельно от Alert Engine — просто список просроченных `reply_sla_deadline`, без изменения состояния

### Проверено
- Синтетика: 3/30дн → yellow, 5/30дн → red, дубль review_id не считается дважды; полный цикл open→update→resolve на изолированной БД
- Реальный датасет (52 отзыва): алертов не поднято — ни один тег не набрал 3+ разных негативных отзыва в одном окне, что честно отражает реальность (нет системных проблем в этой выборке)
- Watchdog нашёл 5 просроченных по SLA — все реальные исторические негативные/нейтральные отзывы без ответа

## [2026-07-06] — Collector: реальный сбор отзывов World Class Тюмень

### Добавлено
- `db.py` — инициализация БД, чекпоинты площадок, идемпотентная запись отзывов (`INSERT OR IGNORE` по `UNIQUE(platform, external_review_id)`)
- `collectors/yandex_maps.py` — сбор через Playwright, парсинг по микроразметке schema.org (`itemprop`), не зависит от локали интерфейса; синтетический ID (хэш автор+дата+начало текста), т.к. Яндекс не отдаёт свой ID отзыва
- `collectors/zoon.py` — нативный ID отзыва из `id="comment<hex>"`; корректно отделяет отзывы верхнего уровня от вложенных официальных ответов клуба (иначе ответ клуба считался бы отдельным "отзывом")
- `collectors/twogis.py` — заготовка, задокументировано ограничение
- `main_collect.py` — точка входа, проходит по `client_config.yaml`, пишет новые отзывы, обновляет чекпоинты

### Проверено на реальных данных
- Яндекс Карты: 50 отзывов собрано (в т.ч. реальный негатив с конкретными жалобами)
- Zoon: 2 отзыва, корректно определён `reply_status` (1 с ответом клуба, 1 без)
- **2ГИС заблокировал сбор (403 antibot)** даже с одиночного запроса — подтверждено дважды. Требует проверки с реального IP VPS/дома перед стартом пилота; если блокирует и там — держим клиенту 2ГИС как "недоступен для автосбора", работаем на Яндекс Картах + Zoon

### Решения
- В обоих коллекторах теперь определяется `reply_status` прямо при сборе (наличие ответа компании в разметке страницы) — Watchdog не нужно ждать следующего цикла, чтобы узнать, что на отзыв уже ответили
- Скроллинг/пагинация (Яндекс — скролл контейнера, Zoon — кнопка "Показать ещё") ограничены `max_scrolls`/`max_clicks`, чтобы не пытаться вычерпать всю историю за один прогон

## [2026-07-06] — Архитектура и стартовые файлы

### Добавлено
- `client_config.yaml` — конфиг пилота: World Class Тюмень (ул. Республики, 245), площадки 2ГИС/Яндекс Карты/Zoon, теги, пороги алертов, SLA на ответ
- `db/schema.sql` — схема БД: locations, location_platforms (курсор сбора), reviews, review_tags (aspect-based тональность на уровне темы, не всего отзыва), tag_dictionary, alert_rules, alerts (персистентное состояние тревоги), tone_profile

### Решения
- Тональность считается на уровне тега (`review_tags.tag_sentiment`), а не всего отзыва — иначе жалоба на конкретную тему теряется внутри общего позитивного отзыва
- Подсчёт статистики для алертов ("N жалоб за период") делает SQL-запрос, не LLM — модель не должна "помнить" числа между вызовами
- Watchdog "нет ответа" и Collector "новые отзывы" — разные, узкие выборки (курсор + короткий список pending), не пересканирование всей истории
- Пороги алертов и SLA — данные в `client_config.yaml`, не код, меняются под регламент клиента без правок
- Тон компании — источник `mined`: документа tone-of-voice у клуба нет, стиль вытягиваем из истории их прошлых ответов на отзывах
- Flamp намеренно не подключаем — дублирует отзывы 2ГИС (общий владелец) и блокирует прямой доступ (403)
