### Найдено
Реальный прод-отзыв (World Class, Евгений В., 1★, 2026-07-16) про постоянные
проблемы с Wi-Fi/связью и бесполезные обращения на ресепшен. Модель вернула теги

тема Wi-Fi/связь физически отсутствовала в словаре — модель силой запихнула её в
два неподходящих существующих тега вместо честного "это новая тема". Жорж: "это 2

- `clients/worldclass/client_config.yaml`: добавлен тег "wifi" (категория
  "оснащение") — Wi-Fi/мобильная связь на территории клуба.
- Усилено описание тега "администраторы" — явно включает случаи отказа помочь по
  факту обращения ("на ресепшене сказали, что не могут ничего сделать").
- Сужено описание тега "информирование" — явно исключает реакцию персонала на уже
  возникшую проблему (это "администраторы"), оставлен только смысл "не

- Рассматривали переименование тега "организация" в "UX" для ясности — ОТМЕНЕНО:
  конфликтует с именем категории "UX" (тег совпал бы с категорией — ровно та

## [2026-07-15] — Golden eval: нашли реальный источник низкого accuracy — метрика, не модель

### Исправлено
- `tests/golden/run_eval.py: _tags_match()` заменена на `_compare_tags()` — метрика
  тегов пересчитана с "весь отзыв идеально или нет" (all-or-nothing per review) на
  per-tag (каждый ожидаемый тег считается отдельно). Жорж прямо спросил: "почему я

  per-tag — реальный recall модели оказался 76-82%, а не 42-58%, которые
  показывала старая метрика. Итоговый accuracy на новой метрике: worldclass 89%,
  daudelsport 87% — вплотную у порога 90%, не в разы ниже, как выглядело раньше.
- per-review метрика оставлена в выводе `run_eval.py` для наглядности ("сколько
  отзывов разобраны идеально"), но больше НЕ используется для решения по порогу.

## [2026-07-15] — Tag + Zone: разделили тему и место в разметке отзывов

### Добавлено
- `review_tags.zone` (nullable) — необязательное место (тренажёрный зал/бассейн/...)
  отдельно от `tag` (темы). Причина: один тег на аспект смешивал ЧТО (например
  "оборудование") и ГДЕ (например "тренажёрный зал") — модель произвольно выбирала
  одно или другое, из-за чего одна тема ("оборудование хромает") размазывалась по
  разным зональным тегам от отзыва к отзыву, и Alert Engine никогда не набирал
  порог по реальной проблеме. Жорж: "оборудование супер в тренажёрке — тег должен
  быть оборудование, зона тренажёрный зал — уточнение, а не замена тега".
- Миграция `core/db.py: _migrate()` — `ALTER TABLE review_tags ADD COLUMN zone TEXT`
  для существующих БД (проверено на реальных БД worldclass и daudelsport).
- `insert_review_tag(...)` — новый keyword-параметр `zone: str | None = None`,
  `get_review_tags(...)` — `zone` в возвращаемом словаре. Alert Engine НЕ тронут:
  `get_negative_tag_events`, `recompute_all`, `get_top_tags_by_sentiment_since`
  продолжают считать строго по `tag`, zone физически не попадает в их SELECT.
- `agents/sentiment_analyst.py` — промпт учит модель разделять тег (тема) и zone
  (место, необязательно), с явным примером на конфликтном кейсе "оборудование в
  тренажёрке" (нашли, что без явного примера-исключения модель по умолчанию
  подменяла тему местом).
- `clients/worldclass/client_config.yaml` — добавлен тег "вентиляция" (было только
  упоминание в описании тега "тренажёрный зал", из-за чего вентиляция вне ТЗ
  галлюцинировалась как новый тег), защитные пометки против дублей по образцу
  уже откалиброванного daudelsport ("гардероб"→раздевалка, "инструктор"→тренеры,
  "зона клуба" — категория, не тег).
- Зональный алерт ("в зоне X стабильно копится негатив независимо от темы" —
  нужен управляющему клуба/клиники для вопроса к супервайзеру зоны) — НЕ
  реализован в этой сессии, осознанно отложен как отдельная задача до появления
  реальных данных zone (см. PLAN.md).

### Эксперимент — не удался, задокументирован
- Пробовали поднять tags accuracy сменой `gpt-4o-mini` → `gpt-4o`, добавлением
  `"reasoning"` в JSON-схему aspects (мини-CoT) и per-client few-shot примерами
  (`clients/<slug>/few_shot_examples.yaml`) — все три ИЗМЕРЕННО ухудшили результат
  на golden eval (72%→61% на изолированном тесте reasoning-поля, 79%→63-68% на
  few-shot). Откачено. Инфраструктура few-shot оставлена (пустая, `[]`) на случай,
  если найдётся формулировка, которая реально поможет — включать только по
  результату eval, не по теории. Подробности и цифры — PLAN.md.

### Текущий baseline (temperature=0, mini, без reasoning/few-shot)
- worldclass: sentiment 100%, tags 42-58% (плавает — `gpt-4o-mini` не полностью
  детерминирована даже при temperature=0, документированное ограничение OpenAI),

- daudelsport: sentiment 89-100%, tags 39-67%, accuracy 71-81%
- Оба ниже порога 90%. Открытый вопрос от Жоржа не закрыт: почему ручной разбор
  (с полным контекстом) даёт правильный результат, а один изолированный вызов

### Решения
- В обоих коллекторах теперь определяется `reply_status` прямо при сборе (наличие ответа компании в разметке страницы) — Watchdog не нужно ждать следующего цикла, чтобы узнать, что на отзыв уже ответили
- Скроллинг/пагинация (Яндекс — скролл контейнера, Zoon — кнопка "Показать ещё") ограничены `max_scrolls`/`max_clicks`, чтобы не пытаться вычерпать всю историю за один прогон

### Добавлено
- `client_config.yaml` — конфиг пилота: World Class Тюмень (ул. Республики, 245), площадки 2ГИС/Яндекс Карты/Zoon, теги, пороги алертов, SLA на ответ

### Решения
- Тональность считается на уровне тега (`review_tags.tag_sentiment`), а не всего отзыва — иначе жалоба на конкретную тему теряется внутри общего позитивного отзыва
- Подсчёт статистики для алертов ("N жалоб за период") делает SQL-запрос, не LLM — модель не должна "помнить" числа между вызовами
- Watchdog "нет ответа" и Collector "новые отзывы" — разные, узкие выборки (курсор + короткий список pending), не пересканирование всей истории
- Пороги алертов и SLA — данные в `client_config.yaml`, не код, меняются под регламент клиента без правок
- Тон компании — источник `mined`: документа tone-of-voice у клуба нет, стиль вытягиваем из истории их прошлых ответов на отзывах
- Flamp намеренно не подключаем — дублирует отзывы 2ГИС (общий владелец) и блокирует прямой доступ (403)
