# CHANGELOG — review-monitor

## [2026-07-09] — Флаг --backfill: защита от массовой рассылки при импорте истории

### Исправлено
- **Инцидент 2026-07-08**: при разборе backfill 2ГИС (см. запись ниже) 90 отзывов Даудель Спорт были вручную долиты в БД через `insert_review_if_new` в обход `main_collect.py`, без пометки `notified_at`. Ближайший плановый прогон `main_notify.py` (19:02 МСК) разослал карточку по каждому — 90 сообщений одним пакетом в Telegram. Причина не специфична для ручного скрипта: обычный `main_collect.py` при ПЕРВОМ запуске на новую локацию (backfill истории) сделал бы то же самое — система не различала "первый крупный импорт" и "обычный ежедневный прогон, где новых отзывов мало"

### Добавлено
- `main_collect.py --backfill` — новый явный флаг. Все отзывы, вставленные в этом запуске, сразу помечаются `notified_at` (через новый параметр `insert_review_if_new(..., skip_notify=True)` в `core/db.py`) — `main_notify.py` их не рассылает. Обычный запуск без флага не изменился
- Чек-лист разворачивания нового клиента (шапка `client_config.yaml` обоих клиентов) и README дополнены явным пунктом: первый сбор истории — только с `--backfill`
- Тест `test_insert_review_skip_notify_marks_notified_immediately` в `tests/test_db.py`

### Проверено
- Локальные БД `clients/worldclass/db/` и `clients/daudelsport/db/` синхронизированы с VPS (были рассинхронизированы после вчерашней ручной работы) — на VPS подтверждено 0 отзывов в очереди `notified_at IS NULL` у обоих клиентов, повторной рассылки не будет

## [2026-07-07] — Деплой на VPS

### Добавлено

### Исправлено
- **Баг парсинга дат, найденный только на сервере**: Python 3.10 (VPS) строже 3.12 (локальная машина) — `datetime.fromisoformat` в 3.10 требует ровно 3 или 6 цифр в дробных секундах, а Apify/2ГИС иногда отдаёт нестандартное число (`.82112` вместо `.821120`). Падало в `agents/alert_engine.py` без перехвата; в `core/db.py` и `agents/sentiment_analyst.py` было тихо (try/except проглатывал и просто не считал такие отзывы — под 3.12 это не проявлялось, но под 3.10 отзыв просто выпадал из подсчёта). Исправлено централизованно — все три места переведены на `core/dates.parse_review_date`, дробная часть паддится/обрезается до 6 знаков перед `fromisoformat`. Баг был в проекте, а не в окружении — апгрейд Python на сервере замаскировал бы его, а не убрал

### Проверено
- Полный цикл прогнан вручную на сервере через systemd (`systemctl start ...service`) — `main_collect.py` (все 3 площадки), `main_analyze.py`, `main_reply.py`, `main_alerts.py` (нашёл реальные 6 просрочек по SLA из бэкфилла), `main_notify.py`, `main_digest.py` — все завершились `status=0/SUCCESS`

## [2026-07-06] — Collector: реальный сбор отзывов World Class Тюмень

### Добавлено
- `db.py` — инициализация БД, чекпоинты площадок, идемпотентная запись отзывов (`INSERT OR IGNORE` по `UNIQUE(platform, external_review_id)`)
- `collectors/yandex_maps.py` — сбор через Playwright, парсинг по микроразметке schema.org (`itemprop`), не зависит от локали интерфейса; синтетический ID (хэш автор+дата+начало текста), т.к. Яндекс не отдаёт свой ID отзыва
- `collectors/zoon.py` — нативный ID отзыва из `id="comment<hex>"`; корректно отделяет отзывы верхнего уровня от вложенных официальных ответов клуба (иначе ответ клуба считался бы отдельным "отзывом")
- `collectors/twogis.py` — заготовка, задокументировано ограничение
- `main_collect.py` — точка входа, проходит по `client_config.yaml`, пишет новые отзывы, обновляет чекпоинты

### Проверено на реальных данных
- Яндекс Карты: 50 отзывов собрано (в т.ч. реальный негатив с конкретными жалобами)
- Zoon: 2 отзыва, корректно определён `reply_status` (1 с ответом клуба, 1 без)
- **2ГИС заблокировал сбор (403 antibot)** даже с одиночного запроса — подтверждено дважды. Требует проверки с реального IP VPS/дома перед стартом пилота; если блокирует и там — держим клиенту 2ГИС как "недоступен для автосбора", работаем на Яндекс Картах + Zoon

### Решения
- В обоих коллекторах теперь определяется `reply_status` прямо при сборе (наличие ответа компании в разметке страницы) — Watchdog не нужно ждать следующего цикла, чтобы узнать, что на отзыв уже ответили
- Скроллинг/пагинация (Яндекс — скролл контейнера, Zoon — кнопка "Показать ещё") ограничены `max_scrolls`/`max_clicks`, чтобы не пытаться вычерпать всю историю за один прогон

## [2026-07-06] — Архитектура и стартовые файлы

### Добавлено
- `client_config.yaml` — конфиг пилота: World Class Тюмень (ул. Республики, 245), площадки 2ГИС/Яндекс Карты/Zoon, теги, пороги алертов, SLA на ответ
- `db/schema.sql` — схема БД: locations, location_platforms (курсор сбора), reviews, review_tags (aspect-based тональность на уровне темы, не всего отзыва), tag_dictionary, alert_rules, alerts (персистентное состояние тревоги), tone_profile

### Решения
- Тональность считается на уровне тега (`review_tags.tag_sentiment`), а не всего отзыва — иначе жалоба на конкретную тему теряется внутри общего позитивного отзыва
- Подсчёт статистики для алертов ("N жалоб за период") делает SQL-запрос, не LLM — модель не должна "помнить" числа между вызовами
- Watchdog "нет ответа" и Collector "новые отзывы" — разные, узкие выборки (курсор + короткий список pending), не пересканирование всей истории
- Пороги алертов и SLA — данные в `client_config.yaml`, не код, меняются под регламент клиента без правок
- Тон компании — источник `mined`: документа tone-of-voice у клуба нет, стиль вытягиваем из истории их прошлых ответов на отзывах
- Flamp намеренно не подключаем — дублирует отзывы 2ГИС (общий владелец) и блокирует прямой доступ (403)
